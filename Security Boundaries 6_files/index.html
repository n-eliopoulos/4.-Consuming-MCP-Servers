Consuming MCP Servers

Module 4: Security Boundaries

Sandboxing and Least Privilege

MCP servers should have minimal permissions. The principle: an MCP server compromise should have blast radius limited to its specific capability.

Least privilege by server

ServerShould haveShould NOT have
market-dataRead access to market data APIsWrite access to trading systems
compliance-docsRead access to rule databaseWrite access to enforcement actions
analyticsRead-only database connectionDDL permissions (CREATE, DROP)

Sandboxing patterns

  • Network isolation: Each server can only reach the systems it needs
  • Filesystem restrictions: No access to files outside the server's working directory
  • Resource limits: CPU, memory, and timeout limits per tool invocation
  • Read-only database connections: Analytics servers should never have write access
  • Separate credentials: Each server gets its own API keys with minimal scopes
Defense in depth: Assume any MCP server can be compromised. If the market-data server is compromised, the attacker should only be able to read market data — not access compliance systems, modify trades, or exfiltrate customer data.
Sandboxing is not about preventing bugs — it's about limiting the damage when something goes wrong. Every MCP server should be designed so that its compromise has the smallest possible blast radius.

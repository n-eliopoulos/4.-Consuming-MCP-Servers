Module 3: Dynamic Tool Loading

Building a Tool Registry

At startup, the agent connects to all configured MCP servers, collects tool definitions, and builds a unified mapping — the tool registry — that becomes the central data structure for dynamic tool loading. Without this single mapping, name collisions across servers (two servers both exposing get_customer_data) produce inconsistent routing, so the registry must resolve them before any tool gets called.

The registry pattern

# Registry structure: tool_name → {server, schema, description}
{
  "get_stock_quote":    {"server": "market-data",     "schema": {...}, "description": "..."},
  "check_compliance":   {"server": "compliance-docs", "schema": {...}, "description": "..."},
  "run_query":          {"server": "analytics",       "schema": {...}, "description": "..."},
}

The registry serves two purposes:

  1. For the LLM: Provides the complete list of available tools with descriptions and schemas
  2. For the agent: Maps tool names back to the server that owns them, enabling correct routing of tool calls
Name collisions: If two servers expose a tool with the same name, the registry must handle it — typically by prefixing with the server name (e.g., market-data.get_report vs. analytics.get_report).

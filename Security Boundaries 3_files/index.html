On-Behalf-Of: Identity Boundaries in MCP

Most MCP servers in production at FINRA receive calls from agents that are acting on behalf of a user. The user authenticated through some front end, the agent picked up the work, and the MCP server is being called as part of that user-initiated chain. This is the dominant pattern. (Server-to-server MCP traffic for batch processing or autonomous workloads exists too — covered separately on the previous slide under mTLS, IAM roles, and API keys.)

For the on-behalf-of case, the server can't just trust whatever bearer token shows up; it has to verify the token, identify the user, and enforce that user's permissions for every tool call.

The token chain at a glance

For a user → agent → MCP server flow, three tokens are typically involved:

  1. User OIDC token (from FIP, scope openid finra_profile). Identifies the user. Issued during the user's login at the front-end.
  2. Service-scoped exchanged token (from FIP, scope like finra_profile ESP). Issued by FIP via token exchange (RFC 8693) using the user token + an actor token from the calling service. Carries user identity but is scoped only for the target MCP server.
  3. Actor token (from FIP, client-credentials grant). Proves the calling service itself is authorized to perform a token exchange. Cached and shared across users.

What the MCP server has to do

  • Verify the signature against FIP's JWKS endpoint. Never trust an unverified JWT.
  • Check the audience claim — the token must have been issued for this server's scope. A token scoped for one MCP server cannot be replayed against a different server.
  • Extract user identity and groups from token claims (or by calling FIP's userinfo endpoint with the access token).
  • Authorize per-tool using the user's groups, not the agent's. The agent inherits user permissions; the server enforces them.

Why the chain matters

PropertyImplication
User identity in every callAudit log shows which user took which action via the agent.
Per-service scopeIf the MCP server is compromised, its token cannot reach other services.
Independent server validationThe agent could be malicious; the MCP server still enforces correctly.
Per-user authorizationThe user's AD groups gate every tool call; access changes when group membership changes.
The full mechanics — actor tokens, the RFC 8693 exchange request shape, two-tier caching, and the per-service registration discipline — live in Security, Governance & Data Pipelines (Track 6 Level 300, module 4: Cross-Service Identity & Token Exchange) — that's the production-engineering home for token-exchange depth. The Building MCP Servers course (Track 4 Level 200) carries the server-side wrapping pattern with a reference callout. This slide is the conceptual foundation: tokens carry identity, scope is per-service, and the server validates everything.
On-behalf-of authorization is what makes user identity a first-class concept in user-facing MCP flows. Without it, a single compromised agent — or a single missing scope check — exposes everything for whichever users routed through it.

Server Discovery

In production, MCP clients find servers through configuration files, not hardcoded addresses. The standard format is mcp.json.

The mcp.json format

{
  "servers": {
    "market-data": {
      "transport": "stdio",
      "command": "python",
      "args": ["-m", "mcp_market_data"],
      "env": {"API_KEY": "${MARKET_DATA_KEY}"}
    },
    "compliance-docs": {
      "transport": "sse",
      "url": "https://compliance-mcp.internal.finra.org/sse",
      "env": {"API_KEY": "${COMPLIANCE_KEY}"}
    }
  }
}

Each server entry specifies:

  • transport — stdio (local process) or sse (remote HTTP+SSE)
  • command/args — for stdio: the command to launch the server process
  • url — for SSE: the server's endpoint
  • env — environment variables (API keys, config) passed to the server
Alternative considered: Hardcode the MCP server command or URL directly in your agent code. This works for one server in one environment and breaks the moment you have a second environment (dev vs. prod), a server moves, or a different team needs to discover the same server. The config file decouples agent code from infrastructure topology, so every team can change their server's location without touching consumers. Hardcoding is acceptable only for a throwaway prototype with one server you control end-to-end.
Open the sandbox to load an mcp.json config with 3 servers, connect to each, and print their capabilities. This is the discovery pattern you'll use in every MCP application.

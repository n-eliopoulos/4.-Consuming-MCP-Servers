Consuming MCP Servers

Module 4: Security Boundaries

Authentication and Authorization

The critical mental model: the server must enforce permissions, not the client. The client can lie. MCP supports transport-level authentication on top of this principle, but transport-level authentication is just the first checkpoint.

Authentication mechanisms

  • API keys — simplest approach, passed as environment variables or headers. Suitable for low-risk, internal service-to-service calls. No user context.
  • OIDC bearer tokens — the production pattern at FINRA. The agent presents a token issued by FIP (FINRA Identity Provider) on the user's behalf. Carries user identity, AD group memberships, and scope claims. Every server call is auditable to a specific user.
  • mTLS — mutual TLS for HTTP+SSE transport. Strongest transport-level auth for service-to-service traffic where no user identity needs to propagate.

Tool-level authorization

Not all authenticated callers should access all tools. A client (or a user-on-whose-behalf the agent is acting) with a market-data:read scope should not be able to call check_compliance on the compliance server.

# Server-side authorization check (inside the MCP server)
def handle_tool_call(tool_name, arguments, caller_scopes, caller_groups):
    required_scope = TOOL_SCOPES[tool_name]      # e.g., "compliance:write"
    required_group = TOOL_GROUPS.get(tool_name)  # e.g., "FINRA_COMPLIANCE_OFFICERS"
    if required_scope not in caller_scopes:
        return error(-32001, f"Unauthorized: requires {required_scope}")
    if required_group and required_group not in caller_groups:
        return error(-32001, f"Unauthorized: requires AD group {required_group}")
    return execute_tool(tool_name, arguments)

For FINRA deployments the caller_scopes and caller_groups come from the OIDC token. Token validation, scope extraction, and group propagation are deeper topics — covered next in this module and in Building MCP Servers (Track 4 Level 200).

Server-side enforcement is non-negotiable. The MCP client is code running on someone else's machine. It can be modified, spoofed, or compromised. The server must validate every request independently — including verifying the token signature, not just trusting the headers.
Open the sandbox to connect to a server with and without API keys, and try invoking a tool your key doesn't authorize. Observe the difference between authentication failures and authorization failures.

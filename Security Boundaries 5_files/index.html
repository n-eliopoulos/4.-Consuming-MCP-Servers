Module 4: Security Boundaries

Input Validation and Prompt Injection

MCP servers receive arguments that originated from LLM output. LLM output can be influenced by prompt injection. This creates a real attack vector: prompt injection → tool injection.

The attack chain

  1. Attacker crafts a prompt injection in user input or retrieved documents
  2. LLM is tricked into generating malicious tool arguments
  3. MCP server receives the malicious arguments and executes them

Defense: server-side input validation

MCP servers must never trust that arguments are safe just because they came through the protocol. Every tool must:

  • Validate all inputs against the inputSchema
  • Apply business rules (e.g., query length limits, allowed characters)
  • Reject inputs matching known attack patterns (SQL injection, command injection)
  • Use parameterized queries, not string interpolation
# Bad: string interpolation
query = f"SELECT * FROM trades WHERE id = '{trade_id}'"

# Good: parameterized query
query = "SELECT * FROM trades WHERE id = ?"
params = [trade_id]
Open the sandbox to see input validation in action. Send a normal query, then a query with SQL injection in the arguments. See the server's validation reject the malicious input.

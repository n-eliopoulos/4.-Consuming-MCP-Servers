Consuming MCP Servers

Module 1: What MCP Solves

MCP Architecture

MCP uses a client-server model with a clear separation of concerns at each layer.

The five layers

  1. Host Application — The AI application the user interacts with (IDE, chatbot, agent). Hosts one or more MCP clients.
  2. MCP Client — Manages the connection to a single MCP server. Handles protocol lifecycle, capability discovery, and message routing.
  3. Transport — The communication channel. Two options: stdio (local servers, stdin/stdout) or HTTP+SSE (remote servers, Server-Sent Events for streaming).
  4. MCP Server — Exposes tools, resources, and/or prompts. Implements the MCP protocol. Translates between MCP messages and the external system's native API.
  5. External System — The actual data source or service: a database, API, file system, or third-party service.

The lifecycle

Client                          Server
  │                               │
  │── initialize ────────────────►│  Protocol version + capabilities
  │◄── initialize response ───────│  Server capabilities
  │                               │
  │── tools/list ────────────────►│  Discover available tools
  │◄── tools list ────────────────│  Tool definitions
  │                               │
  │── tools/call ────────────────►│  Invoke a tool
  │◄── tool result ───────────────│  Result content
  │                               │
  │── shutdown ──────────────────►│  Clean disconnect
  │                               │

The initialize handshake is mandatory. The client declares its protocol version and capabilities; the server responds with its own. This ensures both sides agree on what's supported before any work begins. If version negotiation fails, the connection must close. Attempting to proceed with a mismatched protocol version produces behavior neither side can predict.

One host application can have multiple MCP clients, each connected to a different MCP server. This is how a single agent can access market data, compliance tools, and analytics simultaneously.

Error Handling and Timeouts

Production MCP clients must handle four categories of failure:

1. Transport errors

Server unreachable, network timeout, connection refused. The client can't communicate with the server at all.

2. Protocol errors

Invalid JSON-RPC messages, unsupported methods, malformed requests. The server received the message but can't process it.

3. Tool errors

The tool executed but failed at its task. Response has isError: true with a descriptive message. Example: invalid ticker symbol, unauthorized access.

4. Timeouts

The server is reachable but takes too long to respond. Set timeouts per-tool based on expected latency.

Graceful degradation

If a server is down, the agent should continue with reduced capabilities, not crash. Log the failure, remove the server's tools from the available set, and inform the LLM that certain capabilities are unavailable.

try:
    client.initialize()
except ConnectionError:
    print(f"Server {server_name} unavailable — continuing without it")
    # Remove this server's tools from the agent's tool set
    # The agent continues with remaining servers

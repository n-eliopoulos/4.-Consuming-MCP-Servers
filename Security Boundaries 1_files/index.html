Consuming MCP Servers

Module 4: Security Boundaries

The MCP Trust Model

Every MCP connection crosses a trust boundary. Understanding where these boundaries are, what data crosses them, and what happens when one side is compromised is the foundation of MCP security — because the client cannot verify what the server does with data it receives, and the server cannot verify the client's identity without explicit authentication.

Three trust boundaries

  1. Host ↔ Client: The host application trusts the MCP client it runs. This is an in-process boundary — the client is part of the host.
  2. Client ↔ Server: The client communicates with external MCP servers over a transport (stdio or HTTP+SSE). This is the primary security boundary. The server is external code that may be maintained by a different team or organization.
  3. Server ↔ External System: The MCP server accesses databases, APIs, or services. The server's permissions on the external system determine what it can do.

Trust levels at FINRA

Not all servers are equal:

  • High trust: Internal compliance servers maintained by your team, running in your VPC
  • Medium trust: Internal market data servers maintained by another team
  • Low trust: Community MCP servers or third-party integrations
Key principle: Each boundary is a trust decision. The client can't verify what the server does with the data it receives. The server can't verify that the client is who it claims to be without proper authentication. Security must be enforced at each boundary independently.
Frame security as boundary analysis, not a checklist. For every MCP connection, ask: what data crosses this boundary, who controls each side, and what happens if one side is compromised?

Module 1: What MCP Solves

Why MCP vs. Direct API Integration

MCP is not always the right choice. Understanding the trade-offs helps you make the right architectural decision.

Direct API integration

  • Faster for one-off tools — no protocol overhead, no server to build
  • Full control — custom error handling, custom retry logic, custom auth
  • Tight coupling — changes to the API require changes to the client

MCP integration

  • Standardized discovery — clients discover tools at runtime, not compile time
  • Hot-swappable servers — add/remove capabilities without redeploying the agent
  • Ecosystem reuse — community servers for common integrations (GitHub, Slack, databases)
  • Consistent error handling — JSON-RPC error codes across all integrations

Decision framework

CriterionFavors MCPFavors Direct
ComposabilityMulti-tool agents, dynamic tool setsSingle-purpose integration
DiscoveryTools change over timeFixed, known tool set
ReuseMultiple apps use the same toolsOne app, one tool
PerformanceAcceptable latency budgetSub-millisecond critical path
SimplicityStandardized patternsMinimal moving parts
Use MCP when you need composability, discovery, or multi-tool agents. Use direct integration for single-purpose, performance-critical paths. Open the sandbox to evaluate scenarios against this framework.
Rejected alternative: A shared internal SDK wrapping all direct integrations solves code duplication but not runtime discovery: adding a new tool still requires an agent redeploy. That approach is correct when your tool set is fixed and you control both sides of every integration.

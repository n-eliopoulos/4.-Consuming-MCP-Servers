Routing Tool Calls Back to Servers

When the LLM requests a tool call, the agent must route it to the correct MCP server. The tool registry provides this mapping.

The routing pattern

def handle_tool_call(tool_name, arguments, registry, clients):
    """Route a tool call to the correct MCP server."""
    if tool_name not in registry:
        return {"isError": True, "content": f"Unknown tool: {tool_name}"}

    server_name = registry[tool_name]["server"]
    client = clients[server_name]
    return client.call_tool(tool_name, arguments)

Multi-server conversation flow

  1. User asks: "Check if trade T-2024-0847 complies with Rule 3110, and show me AAPL's current price"
  2. LLM requests check_compliance → registry routes to compliance-docs server
  3. LLM requests get_stock_quote → registry routes to market-data server
  4. Both results feed back to the LLM for a unified response

The LLM doesn't know or care which server owns which tool. It just calls tools by name. The agent handles the routing transparently.

Open the sandbox to extend the agent loop with multi-server routing. The LLM will request tools from different servers in the same conversation, and the agent routes each call correctly.
